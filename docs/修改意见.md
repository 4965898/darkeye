# DarkEye 项目架构修改意见

> 基于对项目整体架构的阅读，提出以下修改建议。**前提**：已接受 UI 直接 SQL 读写，不需要中间层（Service/Repository），保持 `core.database.query/insert/update/delete` 与 `get_connection` 的直连模式。

---

## 一、现状概览

### 1.1 当前架构分层

```
main.py (入口)
├── core/           # 核心业务：爬虫、数据库、图、推荐
├── controller/     # 控制器：信号总线、消息服务、快捷键
├── ui/             # 界面层：页面、组件、导航
├── server/         # 本地 FastAPI 对接浏览器插件
├── darkeye_ui/     # 可复用 UI 组件库
└── cpp_bindings/   # C++ 扩展（力导向图等）
```

### 1.2 数据流（已接受模式）

- **UI → 数据库**：页面/组件直接 `from core.database.query import xxx`，或通过 `SqliteQueryTableModel` / `SqliteEditableTableModel` 使用 `get_connection` 执行 SQL
- **跨线程**：Server (FastAPI) → ServerBridge (Qt Signal) → MainWindow
- **数据变更通知**：`GlobalSignalBus` 发出 `work_data_changed`、`actress_data_changed` 等，页面监听后刷新

### 1.3 文档与实现不一致

- `docs/develop.md` 仍写「Database: SQLite + SQLAlchemy」「逐步迁移到 Repository」
- 实际已统一为 sqlite3 + `get_connection`，无 ORM，`junk/repositories/` 已废弃
- 需同步文档以反映当前架构决策

---

## 二、修改建议（按优先级）

### 1. 文档同步（P0）

**问题**：`develop.md`、`architecture-refactor-proposal.md` 中的描述与当前实现不符。

**建议**：
- 在 `develop.md` 中明确：**数据层采用 sqlite3 + `get_connection`，UI 直接调用 `core.database.query/insert/update/delete`，不引入 ORM 与 Repository 中间层**
- 删除或标注 `architecture-refactor-proposal.md` 中关于 Service 层、Repository 迁移、UI 通过 Service 访问数据等与当前决策不符的建议
- 在 `db-design.md` 或新建 `data-layer.md` 中简要说明：连接策略、query 模块职责、写操作后需 emit 的全局信号

---

### 2. Query 模块组织优化（P1）

**现状**：`core/database/query/` 已按实体拆分为 `work.py`、`actress.py`、`actor.py`、`tag.py`、`dashboard.py`、`statistics.py`、`private.py`，但函数命名风格不统一（如 `findActressFromWorkID` vs `get_actress_info`）。

**建议**：
- 统一命名：读操作用 `get_*` / `query_*` / `fetch_*`，存在性检查用 `exist_*`
- 将 `findActressFromWorkID`、`findActorFromWorkID` 等改为 `get_actress_from_work_id`、`get_actor_from_work_id`（可保留旧名作为 deprecated 别名，逐步迁移调用方）
- 在 `query/__init__.py` 的 docstring 中说明各子模块职责，便于新人理解

---

### 3. 写操作后信号发射规范（P1）✅ 已落实

**现状**：部分 insert/update/delete 函数内部或调用方会 emit `global_signals.*_changed`，但缺乏统一约定，容易遗漏。

**已落实**：
- 在 `core/database/insert.py`、`update.py`、`delete.py` 顶部增加注释：**写操作完成后，调用方负责 emit 对应的 `global_signals.*_changed`**
- 在每个写函数的 docstring 中注明「调用后需 emit xxx 信号」
- 建立「写操作 → 信号」映射表：`docs/write_ops_signal_mapping.md`

---

### 4. Router 瘦身（P1）

**现状**：`router.py` 中 `_handle_work_route` 等逻辑调用了 `get_actor_allname` 等数据访问，Router 承担了部分业务逻辑。

**建议**：
- Router 只负责 `push(route_name, **params)` 与页面切换
- 将「根据 actor_id 获取显示名并填充到页面」等逻辑移至目标页面（如 WorkPage）的 `showEvent` 或初始化逻辑
- 通过 `Router.instance().push("work", actor_id=123)` 传参，由 WorkPage 自行调用 `get_actor_allname` 等

---

### 5. MainWindow 职责拆分（P2）

**现状**：MainWindow 负责布局、路由、状态栏、插件抓取处理、快捷键等，职责较多。

**建议**：
- 抽取 `ApplicationController` 或 `CaptureHandler`：将 `handle_capture_data`、`handle_capture` 等从 MainWindow 移出
- 状态栏相关逻辑封装为 `StatusBarController`，与 MainWindow 解耦
- 页面已通过 Router 懒加载，可保持现状；若启动仍慢，可考虑进一步延迟非首屏页面的创建

---

### 6. 全局信号收敛与文档化（P2）

**现状**：`GlobalSignalBus` 有 10+ 信号，订阅关系分散，数据流难追踪。

**建议**：
- 在 `controller/GlobalSignalBus.py` 中为每个信号添加 docstring：**触发场景、典型订阅者**
- 优先使用父子组件间的局部 Signal，仅在跨模块、跨层级时使用 `global_signals`
- 合并语义接近的信号（如 `actress_data_changed` 与选择器刷新）需评估影响后再动

---

### 7. CrawlerManager 与 DataUpdate 职责拆分（P2）

**现状**：`CrawlerManager` 负责任务队列、多源分发、结果合并、触发写入；`DataUpdate` 负责标签处理、女优/男优插入、图片下载、DB 写入、信号发射。爬虫逻辑与持久化强耦合。

**建议**（在保持 UI 直连 DB 的前提下）：
- 将「多源合并、清洗」从 CrawlerManager 抽成独立模块（如 `DataMerge`），产出标准结构数据
- 将「标签/女优/男优处理、图片下载、DB 写入」从 DataUpdate 抽成 `WorkImport` 或类似模块，供爬虫和手动添加共用
- CrawlerManager 只负责调度、分发、调用 DataMerge 与 WorkImport，不直接操作 DB 细节

---

### 8. Server 与数据库连接统一（P2）

**现状**：`server/app.py` 的 `check_existence` 等可能直接用 sqlite3 连接，与 `get_connection` 策略不一致。

**建议**：
- Server 统一使用 `from core.database.connection import get_connection`
- 只读接口使用 `get_connection(DATABASE, readonly=True)`，避免与 UI 写操作锁冲突
- 共享查询逻辑可放在 `core.database.query` 中，Server 直接 import 调用

---

### 9. 配置模块化（P3）

**现状**：`config.py` 混合路径解析、QSettings、常量，职责略杂。

**建议**：
- 拆分为 `config/paths.py`（路径、打包适配）、`config/settings.py`（QSettings、ini）、`config/constants.py`（版本号、默认值）
- 或保持单文件，但用注释分区，便于后续按需拆分

---

### 10. 启动顺序与错误恢复（P3）

**现状**：`main.py` 顺序执行各阶段，任一步骤失败可能导致半初始化状态。

**建议**：
- 将启动逻辑封装为 `Bootstrap` 或 `ApplicationContext`，各阶段有明确成功/失败状态
- 数据库迁移在创建 MainWindow 之前完成，避免 UI 已展示但 DB 未就绪

---

## 三、不采纳的建议（与当前决策一致）

以下为原 `architecture-refactor-proposal.md` 中的建议，因已接受「UI 直接 SQL，无中间层」而**不采纳**：

| 原建议 | 不采纳原因 |
|--------|------------|
| 引入 Service 层，UI 通过 Service 访问数据 | 已接受 UI 直连 query/insert/update/delete |
| 按实体拆分 Repository，逐步迁移 query 函数 | 已接受过程式函数，不引入 Repository |
| 依赖注入 IWorkService、IActressService | 与直连模式冲突 |
| 将 check_existence 迁移到 WorkRepository/WorkService | 无 Repository/Service，可迁移到 query 模块即可 |

---

## 四、推荐实施顺序

| 阶段 | 内容 | 预期收益 |
|------|------|----------|
| **P0** | 文档同步（develop.md、architecture-refactor-proposal.md） | 避免误导，明确架构决策 |
| **P1** | Query 命名规范、写操作信号规范、Router 瘦身 | 可维护性提升 |
| **P2** | MainWindow 职责拆分、信号文档化、Crawler/DataUpdate 拆分、Server 连接统一 | 结构更清晰，耦合降低 |
| **P3** | 配置模块化、启动封装 | 长期可维护性 |

---

## 五、小结

在「UI 直接 SQL 读写、无中间层」的前提下，架构改进重点在于：

1. **文档与实现一致**：明确数据层策略，移除过时描述
2. **规范与约定**：Query 命名、写操作后信号、Router 职责边界
3. **职责拆分**：MainWindow、Crawler/DataUpdate、Server 连接
4. **可观测性**：信号文档化，便于排查数据流

不引入 Service/Repository 层，保持当前直连模式的简洁性，同时通过上述改进提升可维护性与一致性。
